## 代码规范

### 通用原则
- 写自文档化的代码，不加多余注释。只在复杂业务逻辑、非显而易见的算法或 workaround 处加注释
- Schema-First：优先用 JSON Schema / 类型定义驱动开发，减少手写模型
- 测试先行：改动应有对应测试覆盖
- **SOLID 原则**：
  - **SRP**（单一职责）：每个模块/类/函数只做一件事。函数过长时考虑拆分
  - **OCP**（开闭原则）：通过扩展增加行为，而非修改已有代码
  - **LSP**（里氏替换）：子类/实现必须能无缝替换父类/接口，不破坏契约
  - **ISP**（接口隔离）：接口按职责拆分，不强迫调用方依赖用不到的方法
  - **DIP**（依赖反转）：高层模块不依赖低层模块的具体实现，通过抽象（接口/类型）解耦
  - 具体设计模式选择（Strategy、Factory、Observer 等）根据场景判断，复杂设计决策使用 `/design-guide`
- **模块化清晰**：每个模块职责边界明确，模块间通过接口通信，禁止跨模块直接访问内部实现。目录结构必须反映模块划分
- **方法拆解细致**：复杂逻辑必须拆分为语义明确的小函数，每个函数只做一个层级的事情（编排层调编排，执行层做执行）。禁止"上帝函数"——一个函数包揽所有逻辑

### TypeScript / 前端
- **禁止 `any`**：用 `unknown` + type guard 替代；API 响应用生成的接口类型
- **Import 顺序**：外部库 → 内部绝对路径（generated/、constants/、hooks/）→ 相对路径 → 资源文件（SVG、样式）→ 类型单独分组
- **React**：仅函数组件；Props 用 interface 定义；`useCallback` 处理事件、`useMemo` 处理昂贵计算；`useEffect` 依赖数组完整
- **样式**：优先使用组件库主题 token，不硬编码颜色值；BEM 命名自定义 CSS

### Node.js / Go
- Node.js：async/await 为主，避免回调地狱；错误在边界处理（中间件层），内部用 throw
- Go：error 显式处理，不忽略；struct 组合优于继承

## AI 驱动开发模式

Dorian 采用 AI-native 工作方式，以下规则约束 AI 助手的行为以适配此模式。

### 输出方式
- **CLI-First**：当 Dorian 要求构建工具或功能时，优先提供 CLI 方案，再考虑 UI。CLI 方便 agent 调用和验证
- **迭代不回退**：遇到问题继续修改迭代，不建议 `git revert`，除非 Dorian 明确要求
- **Prompt 简洁时充分推理**：Dorian 的指令可能很短，AI 需要基于项目上下文充分推理意图，不要因为 prompt 短就只做表面工作

### 上下文管理
- **文档驱动**：每个项目维护 docs 文件夹。生成方案或做决策时，优先检查项目 docs 中是否有可复用的已验证方案
- **跨项目复用**：如果其他项目有相似问题的解决方案文档，主动引用
- **关键信息沉淀**：重要的技术决策、架构设计、踩坑记录应建议 Dorian 写入项目 docs，而非只存在于对话中

### Session 管理（强制）

长 session 会导致上下文冗余、幻觉和污染。以下规则适用于所有项目：

- **单 session 单主题**：一个 session 只做一件事（一个功能、一个 bug fix、一项部署）。完成后开新 session
- **跨主题提醒（AI 职责）**：如果 Dorian 在当前 session 中切换到不相关的主题，AI 必须主动提醒："建议开新 session 处理这个话题，避免上下文污染"
- **做完即 commit，不 push**：每完成一个逻辑步骤立即 `git commit`，进度沉淀到 git 而非对话。push 由 Dorian 手动决定
- **代码风格对齐（AI 职责）**：生成代码前先检查项目的 `.prettierrc*` / `.eslintrc*` 配置，按项目规则生成。commit 前必须跑 `pnpm format`（或项目对应的 format 命令）确保风格一致，禁止产生纯风格 diff
- **自查 diff（AI 职责）**：每步代码改动完成后，AI 必须主动跑 `git diff` 自查，确认：(1) 没有改不该改的文件 (2) 没有偏离原始需求 (3) 没有引入多余改动。发现问题自行修正后再提交，不把 review 成本转嫁给 Dorian
- **关键决策写 memory 或文档**：重要的技术决策、踩坑记录不能只存在于对话中，必须持久化到 memory 文件或项目 docs
- **何时必须开新 session**：
  - 当前任务完成并 commit 后
  - 感觉 AI 回答质量下降（重复、遗忘、幻觉）时
  - 需要切换到完全不同的任务时

### 协作模式
- **Dorian 关注架构和决策，AI 负责实现细节**：不需要逐行解释生成的代码，除非 Dorian 问
- **多项目场景**：Dorian 可能同时在多个项目间切换，AI 需要快速理解当前上下文，不混淆项目
