## 代码规范

### 通用原则
- 写自文档化的代码，不加多余注释。只在复杂业务逻辑、非显而易见的算法或 workaround 处加注释
- Schema-First：优先用 JSON Schema / 类型定义驱动开发，减少手写模型
- 测试先行：改动应有对应测试覆盖
- **SOLID 原则**：
  - **SRP**（单一职责）：每个模块/类/函数只做一件事。函数过长时考虑拆分
  - **OCP**（开闭原则）：通过扩展增加行为，而非修改已有代码
  - **LSP**（里氏替换）：子类/实现必须能无缝替换父类/接口，不破坏契约
  - **ISP**（接口隔离）：接口按职责拆分，不强迫调用方依赖用不到的方法
  - **DIP**（依赖反转）：高层模块不依赖低层模块的具体实现，通过抽象（接口/类型）解耦
  - 具体设计模式选择（Strategy、Factory、Observer 等）根据场景判断，复杂设计决策使用 `/design-guide`
- **模块化清晰**：每个模块职责边界明确，模块间通过接口通信，禁止跨模块直接访问内部实现。目录结构必须反映模块划分
- **方法拆解细致**：复杂逻辑必须拆分为语义明确的小函数，每个函数只做一个层级的事情（编排层调编排，执行层做执行）。禁止"上帝函数"——一个函数包揽所有逻辑

### TypeScript / 前端
- **禁止 `any`**：用 `unknown` + type guard 替代；API 响应用生成的接口类型
- **Import 顺序**：外部库 → 内部绝对路径（generated/、constants/、hooks/）→ 相对路径 → 资源文件（SVG、样式）→ 类型单独分组
- **React**：仅函数组件；Props 用 interface 定义；`useCallback` 处理事件、`useMemo` 处理昂贵计算；`useEffect` 依赖数组完整
- **样式**：优先使用组件库主题 token，不硬编码颜色值；BEM 命名自定义 CSS

### Node.js / Go
- Node.js：async/await 为主，避免回调地狱；错误在边界处理（中间件层），内部用 throw
- Go：error 显式处理，不忽略；struct 组合优于继承

## 系统设计规范

规范以 **Google SRE、AWS Well-Architected Framework、12-Factor App、Clean Architecture** 为基线。规则分两级：

- **强制（Invariant）**：标记【强制】的规则在所有项目中不可违反
- **推荐（Default）**：其余为推荐默认值，可根据项目规模和场景调整。简单项目 / 原型期可酌情简化

### 一、架构设计

- **分层架构（强制）**：至少分为 Controller/Handler → Service/UseCase → Repository/DAL 三层。禁止 Controller 直接操作数据库，禁止 Service 层感知 HTTP/RPC 协议细节
- **Bounded Context**：按领域边界划分模块，每个模块拥有独立的数据模型。跨模块通信通过明确的接口/事件，禁止共享数据库表
- **依赖方向单向**：外层依赖内层，内层不感知外层。基础设施（DB、MQ、外部 API）作为可替换的适配器注入
- **配置外置（12-Factor）**：环境相关配置（数据库连接、密钥、Feature Flag）通过环境变量或配置中心注入，禁止硬编码在代码中
- **无状态服务**：业务服务不在本地存储会话状态，状态下沉到 Redis/DB。确保服务可水平扩展
- **故障隔离**：外部依赖调用推荐配置超时和重试（指数退避）；面向用户的关键链路应增加熔断。单个依赖故障不得导致整个系统不可用

### 二、API 设计

- **RESTful 语义**：资源用名词复数（`/users`、`/orders`），操作用 HTTP Method 表达。禁止 `GET /getUser`、`POST /deleteOrder` 这类 RPC 风格 URL
- **版本管理**：API 必须带版本号（URL path `/v1/` 或 Header），破坏性变更必须升版本
- **统一响应格式**：
  ```json
  { "code": 0, "message": "ok", "data": {}, "traceId": "xxx" }
  ```
  错误响应附带机器可读的 error code 和人可读的 message，禁止裸抛堆栈
- **分页标准化**：列表接口必须支持分页，使用 `cursor` 游标分页（大数据量）或 `page/pageSize`（后台管理），返回 `total` 和 `hasMore`
- **幂等设计**：写操作（POST/PUT）必须支持幂等。关键操作使用 idempotency key，防止重复提交
- **入参校验**：API 边界做严格校验（类型、范围、长度、格式），内部方法信任上游数据。校验失败返回 400 + 具体字段错误

### 三、数据库设计

- **范式与反范式平衡**：OLTP 表至少满足 3NF；读多写少的查询场景允许适度冗余，但必须注释说明冗余原因和数据同步策略
- **主键策略**：优先使用自增 bigint 或 Snowflake ID，禁止使用 UUID 作为 InnoDB 聚簇索引主键（随机写入导致页分裂）
- **索引纪律**：
  - 每条慢查询必须有对应索引优化或明确说明为何无法优化
  - 联合索引遵循最左前缀原则，区分度高的列在前
  - 禁止在低基数列（如 `status`、`gender`）上建独立索引
- **变更纪律**：Schema 变更必须通过 Migration 文件管理（版本化、可回滚）。禁止手动在生产环境执行 DDL
- **软删除**：业务数据使用 `deleted_at` 软删除，审计/合规数据永不物理删除
- **大表策略**：预估超过 1000 万行的表，设计之初就规划分区或分表方案

### 四、错误处理与可观测性

- **结构化日志（强制）**：日志必须为 JSON 格式，包含 `timestamp`、`level`、`traceId`、`service`、`message`、`context`。禁止 `console.log("error happened")`
- **错误分级**：
  - `WARN`：可自愈的异常（重试成功、降级生效）
  - `ERROR`：需要人工介入的异常（数据不一致、外部依赖持续失败）
  - `FATAL`：服务不可用，必须立即告警
- **链路追踪**：跨服务调用必须传播 traceId。每个请求从入口到出口可追溯完整调用链
- **健康检查**：每个服务暴露 `/health`（存活探针）和 `/ready`（就绪探针），分别检测进程存活和依赖可用性
- **指标埋点（RED）**：核心接口必须采集 Rate（请求速率）、Errors（错误率）、Duration（延迟分布 P50/P95/P99）

### 五、性能工程

- **缓存策略**：读多写少数据推荐 Cache-Aside 模式；缓存必须设置 TTL，禁止永不过期。高并发场景需考虑防击穿、防穿透、防雪崩策略
- **异步化**：耗时操作（邮件、文件处理、第三方回调）走消息队列异步处理，接口只负责投递任务并返回任务 ID
- **批量优先**：N+1 查询是性能杀手。循环内的 DB/API 调用必须改为批量查询 + 内存关联
- **资源预算**：前端首屏 JS ≤ 200KB（gzipped）、LCP ≤ 2.5s、接口 P95 ≤ 500ms。超标必须优化或给出技术方案

### 六、安全基线

- **输入永不信任**：所有用户输入必须校验 + 转义。SQL 用参数化查询，HTML 用框架自动转义，禁止字符串拼接
- **认证鉴权分离**：认证（你是谁）和鉴权（你能做什么）必须是独立中间件，不混在业务逻辑中
- **最小权限原则**：数据库账号按服务隔离，只授予必要权限。禁止应用使用 root/admin 账号连接生产数据库
- **敏感数据脱敏**：日志、错误响应中禁止输出密码、token、手机号、身份证等敏感信息。存储加密，传输 HTTPS
- **密钥管理**：密钥/凭证通过环境变量或 Vault 注入，禁止提交到代码仓库。`.env` 文件加入 `.gitignore`

### 七、测试策略

- **测试金字塔**：
  - 单元测试（70%）：覆盖核心业务逻辑、边界条件、异常路径
  - 集成测试（20%）：覆盖 API 端到端、数据库交互、外部服务 Mock
  - E2E 测试（10%）：覆盖核心用户流程
- **测试独立性**：每个测试用例可独立运行，不依赖执行顺序。测试数据 setup/teardown 自管理
- **边界覆盖**：空值、空数组、超长字符串、并发、超时——这些边界必须有测试用例

### 八、Git 与发布

- **Commit 规范（Conventional Commits）**：`feat:` / `fix:` / `refactor:` / `chore:` / `docs:` / `perf:` / `test:`，scope 可选。禁止 `update`、`fix bug` 等无意义 message
- **分支策略**：`main`（生产）+ `develop`（开发）+ `feature/*`（特性）+ `hotfix/*`（紧急修复）。Feature 分支生命周期不超过 3 天
- **Code Review 必须**：所有代码通过 PR 合入，至少 1 人 approve。PR 描述必须包含：改了什么、为什么改、如何验证
- **发布纪律**：禁止周五下午发布。上线必须有回滚方案，灰度发布优先

## 深度技术文档规范

当我要求总结或撰写技术深度文档时，遵循以下标准。文档必须 **学术深度与工程实践并重**，不能停留在概念科普层面。

### 核心原则

1. **深度优先**：每个技术点必须追溯到底层实现原理（源码级、算法级、数据结构级），不接受"黑箱式"描述
2. **原理驱动**：对每个原理（论点），必须提供：
   - **论据**：为什么这样设计？学术依据 / 工程权衡是什么？
   - **底层实现**：源码层面怎么做的？关键数据结构和算法是什么？
   - **代码示例**：可运行的代码片段，展示原理的实际体现
3. **生产导向**：每个知识点都要回答"在真实项目中怎么用、会遇到什么问题"
4. **全面性**：覆盖 设计动机 → 底层原理 → 实现细节 → 生产实践 → 局限与坑点 完整链路

### 每个知识点的论证链路

```
论点（What & Why）
  │  这个东西是什么？为什么需要它？解决什么问题？
  ▼
底层原理（How - 学术/源码层面）
  │  核心算法、数据结构、设计模式
  │  源码关键路径（如 React Fiber 树的 beginWork → completeWork）
  │  时间/空间复杂度分析（如适用）
  ▼
论据与权衡（Why this way）
  │  为什么选择这个方案而非其他？
  │  设计权衡（trade-off）是什么？
  │  与同类方案的本质差异
  ▼
代码示例（Proof）
  │  可运行的代码片段，验证原理
  │  标注关键行的作用
  ▼
生产应用（Real-world）
  │  在实际项目中如何落地
  │  配置、调优、最佳实践
  ▼
缺陷与局限（Limitations）
  │  这个方案不适合什么场景？
  │  性能瓶颈在哪？边界条件是什么？
  ▼
坑点（Pitfalls）
    生产上实际遇到过或大概率会遇到的问题
    附带根因分析和解决方案
```

### 文档结构模板

```
# 标题

> **定位**：一句话说明文档范围和深度（如"源码级解析"/"架构设计级"/"工程实践级"）
> **约定**：语言/环境/版本约束（如有）

## 目录（内容较长时必加）

## 一、背景与设计动机
  - 解决什么问题？没有它之前怎么做的？
  - 与同类方案的本质区别（表格对比）

## 二、核心原理与底层实现
  - 关键数据结构 + 算法（配合 ASCII 架构图）
  - 源码关键路径解析（标注文件路径和函数名）
  - 复杂度分析（如适用）

## 三 ~ N-3、深度专题
  - 每个子主题按「论点 → 底层原理 → 代码示例 → 生产应用」展开
  - 代码示例必须可运行或接近可运行，标注关键行

## N-2、对比分析（如适用）
  - 同类方案多维度对比（表格：原理、性能、适用场景、生态）
  - 明确结论：什么场景用什么

## N-1、生产踩坑与最佳实践
  - 坑点：根因分析 + 解决方案 + 预防措施
  - 最佳实践：经过生产验证的配置和模式

## N、面试速答 / 高频问题（如适用）
  - 精炼话术，可直接口述
  - 区分"简答版"和"深度追问版"
```

### 格式约定
- 大章节用中文序号（一二三四五...）
- 副标题用 `——` 分隔（如 `## 三、倒排索引——ES 的灵魂`）
- 对比用表格，流程/架构用 ASCII art
- 代码块标注语言（```typescript / ```go / ```sql）
- 关键术语 **加粗**，重要定义用 `> blockquote`
- 源码引用标注文件路径（如 `react-reconciler/src/ReactFiberBeginWork.js`）
