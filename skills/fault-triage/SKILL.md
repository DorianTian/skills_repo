# Fault Triage — DMP 数据接入问题分析

Dorian 抛出问题 → AI 还原链路 + 总结 → 抛出维度选项让 Dorian 选 → 确认后记录到文档。

AI 负责分析和呈现，归因决策是 Dorian 的。

## 归因维度

### 维度一：故障域（选一个）

| 编号 | 名称 | 范围 |
|------|------|------|
| ❶ | 源端 | 数据源连通性、账号、Schema 变更、源端负载 |
| ❷ | 配置 | URN、ETL 规则、Sink 配置、跨库映射、命名规则 |
| ❸ | 调度 | 任务队列、状态机、锁、并行控制、执行模式 |
| ❹ | 执行 | Flink 作业提交/运行/CDC/Handler/AP 交互 |
| ❺ | 管道&存储 | Kafka Topic + 目标存储（HBase/HDFS/Iceberg/MC/Holo/DLC/MySQL/PG） |
| ❻ | 前端 | UI 交互、状态展示、微前端、BFF 层 |

### 维度二：问题性质（选一个）

| 编号 | 名称 | 判断标准 |
|------|------|----------|
| A | 功能异常 | 跑不通、结果错、逻辑 bug |
| B | 性能问题 | 能跑但慢、资源消耗异常、吞吐不足 |
| C | 体验问题 | 功能正常但用户感知差（仅 ❻ 前端） |
| D | 环境问题 | 发版/环境差异/依赖服务/网络引起 |
| E | 流程问题 | 职责边界不清、该平台做的没做、自动化覆盖不到 |

## 执行流程

### Step 1: 加载领域知识

根据问题涉及的模块，从 project6 memory 中选择性读取：

| 问题涉及 | 读取 memory |
|---------|-------------|
| 表结构、字段、关联 | `project_dmp_schema_overview.md`, `project_dmp_schema_legacy.md`, `project_dmp_schema_services.md` |
| 任务执行、状态、队列 | `project_dmp_task_lifecycle.md` |
| 数据源、账号、FCMP | `project_dmp_source_account.md` |
| 出库、血缘、跨库关联 | `project_dmp_data_lineage.md` |
| URN、ODS 命名 | `project_dmp_urn.md`, `project_dmp_ods_naming.md` |
| 跨服务调用 | `work_api_chains.md` |
| 业务主体 | `project_dmp_entity_naming.md` |

同时读取 `pj/docs/data-ingestion-fault-taxonomy.md` 确认最新维度定义。

### Step 2: 还原问题链路

- **截图/报错** → 追踪代码找到报错来源，还原调用链
- **聊天记录/工单** → 提取 who/what/when/where
- **信息不足** → 向 Dorian 追问，不猜测

输出一段简明的**问题链路总结**，说清楚发生了什么、涉及哪些模块和代码路径。

### Step 3: 交互式归因选择

分析完成后，用以下格式呈现，让 Dorian 直接选：

```
📋 **问题链路**
[一段话总结问题的完整链路]

---

**故障域（选一个）**:
1. ❶ 源端 — [为什么可能是这个]
2. ❸ 调度 — [为什么可能是这个]

**问题性质（选一个）**:
1. A 功能异常 — [为什么可能是这个]
2. E 流程问题 — [为什么可能是这个]

**子分类建议**: SC-2 / SC-5 / ...（从文档中匹配）

boss 选个编号就行，比如"❸ + A"或者"2+1"
```

**格式要求**：
- 只列有可能的选项，明显不靠谱的不列
- 每个选项给一句话理由，简洁
- 如果某个维度只有一个合理选项，也要列出来让 Dorian 确认，不要自动选定
- 最后用轻松的方式提示 Dorian 选择，降低选择成本

### Step 4: 确认后记录

Dorian 选完后，将结果追加到 `pj/docs/data-ingestion-fault-taxonomy.md`「问题日志」section（`<!-- 新问题插入在这行下方 -->` 之后），时间倒序：

```markdown
### [简明问题标题]

- **日期**: YYYY-MM-DD
- **归因**: [故障域]-[问题性质]（如 ❸-A）
- **子分类**: 问题 ID + 名称
- **影响范围**: 受影响的任务/表/业务
- **现象描述**: 用户看到了什么
- **根因分析**: 完整链路还原
- **解决方式**: 怎么修的（未解决写「待定」）
- **后续 action**: 防复发措施（可选）
```

Dorian 后续补充根因或解决方式时，更新已有记录。

### Step 5: 无法匹配时

问题不在现有维度内 → 不硬塞 → 告诉 Dorian 并建议新维度 → 确认后更新文档和 memory。

## 注意事项

- 多个问题逐个独立分析，每个单独走一遍选择流程
- 分析要深入追代码，不要只看表面
- **永远不替 Dorian 做归因决定**，只呈现选项
