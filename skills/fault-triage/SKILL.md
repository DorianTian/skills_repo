# Fault Triage — DMP 数据接入问题分析

当 Dorian 描述数据接入相关的问题时，负责**还原问题链路和提供归因维度选项**，由 Dorian **人工拍板最终归因**，然后记录到文档。

AI 不做归因判断，只做分析和维度呈现。归因需要业务上下文和经验判断，这是 Dorian 的决策。

## 归因维度（供参考）

### 维度一：故障域 — 问题出在哪一层

| 层 | 名称 | 范围 |
|----|------|------|
| ❶ | 源端 | 数据源连通性、账号、Schema 变更、源端负载 |
| ❷ | 配置 | URN、ETL 规则、Sink 配置、跨库映射、命名规则 |
| ❸ | 调度 | 任务队列、状态机、锁、并行控制、执行模式 |
| ❹ | 执行 | Flink 作业提交/运行/CDC/Handler/AP 交互 |
| ❺ | 管道&存储 | Kafka Topic + 目标存储（HBase/HDFS/Iceberg/MC/Holo/DLC/MySQL/PG） |
| ❻ | 前端 | UI 交互、状态展示、微前端、BFF 层 |

### 维度二：问题性质 — 什么类型的问题

| 代号 | 名称 | 判断标准 |
|------|------|----------|
| A | 功能异常 | 跑不通、结果错、逻辑 bug |
| B | 性能问题 | 能跑但慢、资源消耗异常、吞吐不足 |
| C | 体验问题 | 功能正常但用户感知差（仅适用于 ❻ 前端） |
| D | 环境问题 | 发版/环境差异/依赖服务/网络基础设施引起 |
| E | 流程问题 | 职责边界不清、该平台做的没做、该用户做的没引导、自动化覆盖不到 |

## 执行流程

### 1. 加载上下文

**归因框架**：读取 `pj/docs/data-ingestion-fault-taxonomy.md`，确认当前的维度定义和子分类。框架随时间演进，始终以文档为准。

**领域知识**：根据问题涉及的模块，从 project6 的 memory 中选择性读取相关文件：

| 问题涉及 | 读取 memory |
|---------|-------------|
| 表结构、字段、关联关系 | `project_dmp_schema_overview.md`, `project_dmp_schema_legacy.md`, `project_dmp_schema_services.md` |
| 任务执行、状态、队列 | `project_dmp_task_lifecycle.md` |
| 数据源、账号、FCMP | `project_dmp_source_account.md` |
| 出库、血缘、跨库关联 | `project_dmp_data_lineage.md` |
| URN、ODS 命名 | `project_dmp_urn.md`, `project_dmp_ods_naming.md` |
| 跨服务调用、API 链路 | `work_api_chains.md` |
| 业务主体、entity 命名 | `project_dmp_entity_naming.md` |

### 2. 深入分析问题

不要看到报错就直接贴标签。好的分析需要理解问题全貌：

- **看到截图/报错信息** → 去代码里找到产生这个报错的具体位置，追踪调用链路
- **看到聊天记录/工单** → 提取关键信息（who/what/when/where），识别问题本质
- **信息不足** → 向 Dorian 追问，明确列出需要什么信息，不猜测

分析要回答两个问题：
1. **发生了什么**：还原问题的完整链路（从用户操作到最终报错）
2. **涉及哪些模块**：代码路径、表、服务、状态流转

### 3. 提供维度选项（不做归因判断）

分析完成后，列出这个问题**可能涉及的维度选项**，让 Dorian 选择：

```
**问题还原**: [简要描述问题链路]

**可能的故障域**:
- ❸ 调度: [理由]
- ❹ 执行: [理由]

**可能的问题性质**:
- A 功能异常: [理由]
- E 流程问题: [理由]

boss 你看归到哪个？
```

列出候选项时给出各自的理由，但不替 Dorian 做决定。如果某个维度明显不靠谱就不用列。

### 4. Dorian 拍板后记录到文档

Dorian 确认归因后，将结果追加到 `pj/docs/data-ingestion-fault-taxonomy.md` 的「问题日志」section（`<!-- 新问题插入在这行下方 -->` 标记之后），时间倒序排列：

```markdown
### [简明问题标题]

- **日期**: YYYY-MM-DD
- **归因**: [故障域]-[问题性质]（如 ❺-B）
- **子分类**: 问题 ID + 名称
- **影响范围**: 受影响的任务/表/业务
- **现象描述**: 用户看到了什么
- **根因分析**: 完整的链路还原和根因解释
- **解决方式**: 怎么修的（未解决写「待定」）
- **后续 action**: 防复发措施（可选）
```

如果 Dorian 后续补充了根因或解决方式，更新已有记录而不是新增条目。

### 5. 无法匹配时

如果问题不在现有维度内，不要硬塞：
1. 告诉 Dorian 现有框架覆盖不了这个问题
2. 建议可能的新维度（说明为什么现有的不够用）
3. Dorian 确认后同时更新 `data-ingestion-fault-taxonomy.md` 和 memory 中的 `workflow_fault_taxonomy.md`

## 注意事项

- 多个问题一次性丢过来时，逐个独立分析
- 分析的核心价值是还原链路，不是贴标签 — 宁可多花时间追代码
- **永远不替 Dorian 做归因决定**，只提供维度选项和分析依据
