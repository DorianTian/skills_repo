# Fault Triage — 数据接入问题归因

## 触发条件

当 Dorian 描述任何与数据接入相关的问题时自动执行。包括但不限于：
- 直接的故障描述（"某张表数据没到"、"Flink 挂了"、"队列卡住了"）
- Oncall 工单内容
- 用户反馈的异常
- 性能相关的抱怨（"出库太慢"、"延迟高"）

## 执行流程

### Step 1: 读取归因框架

读取 `pj/docs/data-ingestion-fault-taxonomy.md` 获取最新的归因维度和子分类定义。

### Step 2: 归因分析

用两个正交维度定位问题：

**维度一：故障域（哪一层）— 互斥必选一**

| 层 | 名称 |
|----|------|
| ❶ | 源端 Source |
| ❷ | 配置 Configuration |
| ❸ | 调度 Scheduling |
| ❹ | 执行 Flink Runtime |
| ❺ | 管道&存储 Pipeline & Storage |
| ❻ | 前端 Frontend & UX |

**维度二：问题性质（什么类型）— 互斥必选一**

| 代号 | 名称 |
|------|------|
| A | 功能异常（跑不通/结果错/逻辑 bug） |
| B | 性能问题（能跑但慢/资源异常） |
| C | 体验问题（功能正常但用户感知差，仅 ❻ 前端） |
| D | 环境问题（配置/发版/依赖服务/网络引起） |

**归因规则：**
- 一个问题只落一个格子，不允许多标签
- 按根因归，不按表现归
- 环境问题(D)归到受影响的故障域
- 信息不足时向 Dorian 追问，不猜测

### Step 3: 输出归因结果

用以下格式输出，简洁直接：

```
**归因**: [故障域]-[问题性质]（如 ❺-B）
**子分类**: 问题 ID + 名称（如 PS-3 HBase Region 热点）
**判断依据**: 一句话说明为什么这么归
```

如果信息不够做出准确归因，列出需要补充的信息。

### Step 4: 追加到文档

将归因结果按以下模板追加到 `pj/docs/data-ingestion-fault-taxonomy.md` 的「问题日志」section（`<!-- 新问题插入在这行下方 -->` 之后），时间倒序：

```markdown
### [问题标题]

- **日期**: YYYY-MM-DD
- **归因**: [故障域]-[问题性质]
- **子分类**: 问题 ID
- **影响范围**: 受影响的任务/表/业务
- **现象描述**: 用户看到了什么
- **根因分析**: 实际发生了什么
- **解决方式**: 怎么修的（如已知）
- **后续 action**: 需要做什么防止复发（可选）
```

如果问题刚发生还没解决，解决方式和后续 action 标注「待定」。

### Step 5: 无法匹配处理

如果问题无法归入现有维度，**不要硬塞**，而是：
1. 告诉 Dorian 这个问题不在现有框架内
2. 建议可能的新维度
3. Dorian 确认后更新 `data-ingestion-fault-taxonomy.md` 和 memory

## 注意事项

- 归因要快，不要长篇大论分析。结论先行，依据一句话
- 如果 Dorian 一次性丢多个问题，逐个归因，每个独立记录
- 归因完直接更新文档，不需要 Dorian 确认"要不要写入"
- 如果 Dorian 后续补充了根因或解决方式，更新已有记录而非新增
