# Fault Triage — DMP 数据接入问题归因

当 Dorian 描述数据接入相关的问题时，用两个正交维度快速定位问题，然后深入分析根因，最后记录到项目文档。

归因的价值在于：积累足够多的归因记录后，能看出哪些模块是高频故障域、哪些类型的问题反复出现，从而指导架构改进和流程优化。所以每次归因都要认真做，不能敷衍。

## 归因维度

### 维度一：故障域 — 问题出在数据接入链路的哪一层

| 层 | 名称 | 范围 |
|----|------|------|
| ❶ | 源端 | 数据源连通性、账号、Schema 变更、源端负载 |
| ❷ | 配置 | URN、ETL 规则、Sink 配置、跨库映射、命名规则 |
| ❸ | 调度 | 任务队列、状态机、锁、并行控制、执行模式 |
| ❹ | 执行 | Flink 作业提交/运行/CDC/Handler/AP 交互 |
| ❺ | 管道&存储 | Kafka Topic + 目标存储（HBase/HDFS/Iceberg/MC/Holo/DLC/MySQL/PG） |
| ❻ | 前端 | UI 交互、状态展示、微前端、BFF 层 |

### 维度二：问题性质 — 这是什么类型的问题

| 代号 | 名称 | 判断标准 |
|------|------|----------|
| A | 功能异常 | 跑不通、结果错、逻辑 bug — 代码或数据层面的错误 |
| B | 性能问题 | 能跑但慢、资源消耗异常、吞吐不足 |
| C | 体验问题 | 功能正常但用户感知差（仅适用于 ❻ 前端） |
| D | 环境问题 | 发版/环境差异/依赖服务/网络基础设施引起 |
| E | 流程问题 | 职责边界不清、该平台做的没做、该用户做的没引导、自动化覆盖不到需要人工兜底 |

### 归因 = 一个坐标点

每个问题落在矩阵中的一个格子，格式为 `❶-A`、`❺-B`、`❶-E` 等。

**关键规则：**
- **互斥**：一个问题只落一个格子
- **按根因归**：表现在前端但根因是后端 → 归到后端对应层
- **环境问题(D)归到受影响的故障域**：AP 发版导致 Flink 跑不起来 → `❹-D`
- **流程问题(E)要识别职责归属**：云 DB 白名单该平台自动化但没覆盖到 → `❶-E`

## 执行流程

### 1. 加载上下文

**归因框架**：读取 `pj/docs/data-ingestion-fault-taxonomy.md`，确认当前的维度定义和子分类。框架随时间演进，始终以文档为准。

**领域知识**：根据问题涉及的模块，从 project6 的 memory 中选择性读取相关文件（不需要全部加载）：

| 问题涉及 | 读取 memory |
|---------|-------------|
| 表结构、字段、关联关系 | `project_dmp_schema_overview.md`, `project_dmp_schema_legacy.md`, `project_dmp_schema_services.md` |
| 任务执行、状态、队列 | `project_dmp_task_lifecycle.md` |
| 数据源、账号、FCMP | `project_dmp_source_account.md` |
| 出库、血缘、跨库关联 | `project_dmp_data_lineage.md` |
| URN、ODS 命名 | `project_dmp_urn.md`, `project_dmp_ods_naming.md` |
| 跨服务调用、API 链路 | `work_api_chains.md` |
| 业务主体、entity 命名 | `project_dmp_entity_naming.md` |

这些 memory 提供了 DMP 平台的架构、表结构、调用链路等领域知识，有了这些上下文才能准确追踪问题链路和判断根因。

### 2. 深入分析再归因

不要看到报错就直接贴标签。好的归因需要理解根因：

- **看到截图/报错信息** → 去代码里找到产生这个报错的具体位置，追踪调用链路
- **看到聊天记录/工单** → 提取关键信息（who/what/when/where），识别问题本质
- **信息不足** → 向 Dorian 追问，明确列出需要什么信息，不猜测

归因分析要回答三个问题：
1. **发生了什么**：还原问题的完整链路（从用户操作到最终报错）
2. **为什么发生**：根因是什么，区分表象和本质
3. **该归到哪**：基于根因选择故障域和问题性质

### 3. 输出归因结果

结论先行，分析在后：

```
**归因**: ❶-E（源端 × 流程问题）
**子分类**: S-2 账号
**判断依据**: FCMP big_data_auth 只做 MySQL GRANT，云 DB 的网络白名单需要 fservice db 中间件手动处理，平台自动化没覆盖到

[然后展开详细的链路分析...]
```

### 4. 记录到文档

将归因结果追加到 `pj/docs/data-ingestion-fault-taxonomy.md` 的「问题日志」section（`<!-- 新问题插入在这行下方 -->` 标记之后），时间倒序排列：

```markdown
### [简明问题标题]

- **日期**: YYYY-MM-DD
- **归因**: [故障域]-[问题性质]（如 ❺-B）
- **子分类**: 问题 ID + 名称
- **影响范围**: 受影响的任务/表/业务
- **现象描述**: 用户看到了什么
- **根因分析**: 完整的链路还原和根因解释
- **解决方式**: 怎么修的（未解决写「待定」）
- **后续 action**: 防复发措施（可选）
```

归因完直接写入文档，不需要问 Dorian "要不要记录"。如果 Dorian 后续补充了根因或解决方式，更新已有记录而不是新增条目。

### 5. 无法匹配时

如果问题不在现有维度内，不要硬塞：
1. 告诉 Dorian 现有框架覆盖不了这个问题
2. 建议可能的新维度（说明为什么现有的不够用）
3. Dorian 确认后同时更新 `data-ingestion-fault-taxonomy.md` 和 memory 中的 `workflow_fault_taxonomy.md`

## 注意事项

- 多个问题一次性丢过来时，逐个独立归因，每个单独记录
- 归因的核心价值是深入分析，不是快速贴标签 — 宁可多花时间追代码也不要浅层归因
- 如果同一个问题涉及多层（比如配置错导致 Flink 挂），按根因归（❷ 配置），不按表象归（❹ 执行）
